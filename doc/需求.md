AI 工具导航站产品需求文档 (PRD)
面向开发与设计的全栈执行规范


1. 项目概述与范围
  本章节明确 AI 工具导航站的核心定位、目标用户群体、项目边界以及分阶段的交付策略。作为一份面向设计、研发与测试团队的执行文档，本 PRD 旨在消除需求歧义，确保 V1 版本的顺利上线及后续版本的有序迭代。
2. 前台功能需求 (Frontend)
  前台页面设计应遵循 “极简主义”，去除所有干扰用户获取信息的元素。页面加载速度是第一优先级。
2.1 1. 首页 (Home)
  首页承担核心流量分发功能，由 “全局导航”、“推荐工具” 与 “最新收录” 三大板块构成。严禁展示 “热门榜单” 或 “浏览量” 数据。
2.1.1 1.1 全局导航 (Global Header)
  全站通用的顶部导航栏。
  - Logo：点击返回首页。
  - 分类链接：横向展示前 5-7 个核心分类，超出部分收入 “更多” 下拉菜单。数据来源于 categories 表。
  - 搜索框：右侧常驻，支持回车跳转搜索页。
2.1.2 1.2 推荐工具区 (Featured Tools)
  该区域完全由管理员在后台控制，用于引导流量至高质量工具。
模块
逻辑与规则
数据源
查询 tools 表，筛选 recommended=1 且 status='published'。排序建议按后台指定的 sort_order 或 created_at DESC。
展示内容
Logo、名称、一句话简介 (short_description)、前 2 个标签 (Tags)。
数量限制
固定展示 8-12 个。若推荐工具不足，不自动补全，按实际数量展示。
2.1.3 1.3 最新收录 (New Arrivals)
  展示站点最新入库的工具，体现内容鲜活度。
  - 逻辑：查询 tools 表，按 created_at DESC 倒序排列。取 Top 10。
  - 样式：与推荐区保持一致的卡片样式。
2.2 2. 分类列表页 (Category)
  URL: /category/:slug。包含侧边栏筛选器与主列表。
2.2.1 2.1 筛选与排序

2.2.2 2.2 列表交互
  - 分页：每页 20 条。底部展示标准分页器 ([1] [2] ... [Next])。
  - 空状态：若筛选组合无结果，展示 “暂无相关工具” 插画，并提供 “重置筛选” 按钮。
2.3 3. 工具详情页 (Detail)
  URL: /tool/:slug。本页面是 SEO 权重的核心承载页。
2.3.1 3.1 核心信息区
  - 头部 (Header)：展示 Logo、工具名称、所属分类 (面包屑)、标签组。
  - 行动点 (CTA)：提供 “访问官网” 按钮。
    - 点击行为：在新标签页打开 website_url。
    - SEO 约束：必须添加 rel="nofollow" 属性，防止权重流失。
    - 无统计：不触发任何 click tracking 请求。
  - 内容主体 (Body)：渲染 Markdown 格式的详细介绍 (description)。允许包含 H2/H3 标题与列表。
2.3.2 3.2 同类推荐 (Related Tools)
  位于页面底部，用于增加内链密度。
  - 筛选逻辑：同一 category_id 下的其他工具。
  - 排序逻辑：优先展示 recommended=1 的精选工具；若不足，按 created_at DESC 补足。排除当前工具 ID。
  - 数量：固定 6 个。
2.4 4. 搜索页 (Search)
  URL: /search?q={keyword}。
  - 匹配范围：工具名称 (name)、短简介 (short_description)。
  - 算法：简单的数据库模糊查询 (LIKE %keyword%)。
  - 排序：匹配结果按 recommended DESC, created_at DESC 排序。
  - 空结果：若无匹配，展示 “未找到相关工具”，并兜底展示 “推荐工具” 列表，避免页面留白。
3. 后台功能需求 (Backend)
3.1 1. 基础管理功能
  后台仅包含对三个核心数据实体的管理：工具、分类、标签。所有页面均需通过 Session/Token 鉴权。
3.1.1 1.1 工具管理 (Tool CRUD)
  核心内容录入界面。包含列表页与编辑页。
  - 列表页：按创建时间倒序。支持按名称搜索。每一行提供 [编辑] [删除] 按钮。
  - 删除保护：物理删除前必须弹出二次确认框。
  - 必填校验：保存时必须校验 Name, Slug, Website_URL 不为空。
3.1.2 1.2 分类与标签管理
  - 分类 (Categories)：支持修改名称与排序 (Sort Order)。删除分类时，若该分类下有关联工具，应提示报错，禁止删除。
  - 标签 (Tags)：平铺式管理。仅支持增删改名称。

7. SEO 规范 (强制执行)
  SEO 是本项目的生命线。以下规则为开发硬性要求，测试验收时必须查看页面源代码验证。
7.1 Meta Tags 生成规则
页面
Title 模板
Description 模板
分类页
{分类名}工具推荐 - 免费{分类名}工具大全 | {站点名}
示例：AI 写作工具推荐 - 免费 AI 写作工具大全 | AI 导航
收录最新{分类名}工具，提供{分类名}软件下载、使用指南与评测。包含免费、中文、开源等精选资源。
工具页
{工具名}介绍 - {一句话卖点/分类名} | {站点名}
示例：ChatGPT 介绍 - 最强 AI 聊天工具 | AI 导航
自动截取 description 字段前 150 字。必须去除 HTML 标签、换行符，并追加 ...。
7.2 Technical SEO
  - Sitemap.xml：每日自动生成。必须包含所有分类页 URL 和所有 status='published' 的工具页 URL。
  - Robots.txt：允许爬虫访问所有公开页面，屏蔽 /admin 和 /api 路径。
  - Canonical：所有页面必须包含 <link rel="canonical" href="..." />，指向当前页面的标准 URL（去除无关 query 参数如广告来源 utm 等，但保留分页和筛选关键参数）。
  - Alt Text：所有工具 Logo 必须包含 alt="{工具名} Logo"。
8. 版本计划与交付清单
V1 核心发布 (MVP)
聚焦于 “纯净内容展示” 与 “SEO 基础建设”。
  - 前台：首页、分类页、详情页、搜索页。
  - 后台：基础 CMS (工具/分类/标签 CRUD)。
  - 质量：LCP < 2.5s，移动端适配，Sitemap 自动生成。
后续维护迭代 (Maintenance)
V1 上线后的体验微调，不涉及新业务模块。
  - 搜索优化：引入分词搜索或搜索结果高亮。
  - 运营效率：后台列表支持更多筛选维度。
  注：本项目暂无引入用户系统、投稿系统或广告系统的计划。后续迭代将严格围绕 “提升内容获取效率” 展开。
9. 非功能需求 (NFR)
9.1.1 网站质量验收清单 (NFR)
测试验收需使用 Lighthouse 和 Chrome DevTools 进行。
9.1.1.1 1. 性能指标 (Performance)
- LCP (最大内容渲染) < 2.5s。
- CLS (累积布局偏移) < 0.1。
9.1.1.2 2. 内容质量 (Content Quality)
- 完整率：所有上线工具的字段完整率必须为 100%。
- 死链率：站内链接 404 率必须为 0%。
9.1.1.3 3. SEO 基础
- Meta：Title/Description 均按规则生成。
- Sitemap：XML 文件可访问且内容最新。
